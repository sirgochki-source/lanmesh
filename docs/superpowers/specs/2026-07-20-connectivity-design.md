# Подпроект A — «Связность»

Дата: 2026-07-20. Статус: дизайн утверждён, план не написан.

## Контекст

Главная слабость lanmesh против Radmin VPN — не крипта и не UX, а **ретрансляция**:
пара, не пробившая NAT, идёт через единственный сервер на белом адресе, и его
полоса, география и аплинк становятся общим узким местом. Подпроект A не трогает
ретранслятор — он уменьшает долю пар, которым ретранслятор вообще нужен.

Подпроект A — первый из трёх. Дальше отдельными циклами: **C** — анонс игровых
серверов пиров (подробный режим GUI), **B** — forward secrecy и идентичность на
паре ключей (ломает совместимость, поэтому последним).

## Область

Входит:

1. Проброс порта через роутер: PCP → NAT-PMP → UPnP-IGD.
2. Транспорт IPv6 (dual-stack сокет).
3. Транзитивный PEX — обмен адресами пиров между пирами.
4. Кэш endpoint'ов на диске между запусками.
5. **Фиксированный локальный UDP-порт** — не заявлялся изначально, обнаружен как
   предусловие для 1 и 4 (см. «Решения»).

Не входит и остаётся на IPv4 сознательно — это свойство протоколов, а не долг:

- **DHT** — компактный формат пиров BEP-5 адресует только IPv4.
- **Ретранслятор** — сервер живёт на белом IPv4.
- **STUN** — для IPv6 не нужен: NAT нет, глобальный адрес интерфейса и есть
  внешний адрес.

## Архитектура

Единственная идея, объединяющая все пять пунктов: у узла есть набор кандидатов на
*свой* адрес и набор кандидатов на адрес *каждого пира*. Подпроект не добавляет
новых механизмов соединения — он добавляет **новые источники кандидатов** и один
новый транспорт.

| | сегодня | добавляем |
|---|---|---|
| свои адреса | STUN, локальные интерфейсы v4 | проброшенный порт от роутера, глобальные IPv6 |
| адреса пиров | сигналка, DHT, peer-reflexive (`FrameAddr`) | PEX от пиров, кэш прошлой сессии |

**Ключевое решение, защищающее от техдолга:** все новые источники вливаются в
**уже существующие точки входа** — `Engine.SetSelfCandidates` для своих,
`Engine.AddProbes` для чужих. Ни один не получает собственной дороги в движок.
Движок по-прежнему не знает, откуда взялся адрес: у него один механизм пробития и
одно правило признания пира — кадр расшифровался ключом сети. Поэтому четыре фичи
не превращаются в четыре параллельные подсистемы.

`AddProbes` для этого готов уже сейчас (`internal/peer/engine.go:336`): принимает
голые адреса без PeerID, дедуплицирует по строке, держит потолок `maxProbes`, не
сбрасывает backoff известным адресам и намеренно отделён от `endpoints` пиров
(потолок `maxCandidates` = 12), чтобы мусор не вытеснял рабочие адреса.

### Транспорт: dual-stack + `netip.AddrPort`

Рассмотренные варианты:

- **Два сокета (v4 + v6) рядом.** Минимальный риск релиза — рабочий v4-путь не
  трогается. **Отвергнут:** оставляет два параллельных транспортных пути навсегда,
  каждая последующая фича обязана уметь оба, и часть из них про v6 забудет.
- **Один dual-stack сокет без миграции типов.** Дёшево. **Отвергнут:**
  IPv4-mapped адреса (`::ffff:1.2.3.4`) расползлись бы во всё, что зовёт `To4()` —
  DHT-анонс, `localEndpoints`, `sameAddr`, `matchSTUN`. Скрытый долг.
- **Один dual-stack сокет + миграция транспортных адресов на `netip.AddrPort`.**
  Принят.

Почему принятый вариант не создаёт долга, а снимает его:

- `netip.Addr` умеет `Is4In6()`/`Unmap()` — нормализация становится **гарантией
  типа на границе сокета**, а не россыпью `To4()` по коду.
- `ReadFromUDPAddrPort`/`WriteToUDPAddrPort` (Go 1.18+; в проекте Go 1.26) **не
  аллоцируют** на пакет, в отличие от `*net.UDPAddr`. Горячий путь ускоряется —
  это выигрыш, а не плата за IPv6.
- Проект уже идёт туда: `netip` используется в `proto.VirtualIP`, `Session.selfIP`,
  `internal/tun`. Транспорт — последний слой на старых типах.

Сокет: `net.ListenUDP("udp", &net.UDPAddr{Port: p})` — Go при неуказанном IP сам
ставит `IPV6_V6ONLY=0`. Фолбэк на `"udp4"`, если не вышло.

**Правило нормализации, отдельно на чтение и на запись.** «Unmap на границе»
описывает только половину:

- **Чтение:** `Unmap()` сразу после `ReadFromUDPAddrPort`. Внутри движка
  mapped-адресов не существует по типу, а не по договорённости — сравнение,
  дедуп и хранение работают с чистым v4.
- **Запись:** адрес отдаётся в `WriteToUDPAddrPort` **как есть, unmapped**.
  Отображение в v4-in-v6 при формировании sockaddr для AF_INET6-сокета
  выполняет стандартная библиотека.

**Проверено эмпирически** (Go 1.26.5, Windows 11, 2026-07-20) — это был линчпин
всего выбора транспорта, поэтому проверялся до написания плана, а не по ходу:

```
udp4-слушатель: 127.0.0.1:58425      dual-stack сокет: [::]:58463 (IP=::)

OK  доставка unmapped v4 — получено "unmapped" от 127.0.0.1:58463
OK  доставка v4-in-v6 — получено "mapped"
    исходный вид адреса: ::ffff:127.0.0.1 (Is4=false Is4In6=true), после Unmap: 127.0.0.1
OK  v4 -> dual-stack доставлено
OK  Unmap даёт чистый v4 — 127.0.0.1
OK  полный цикл запрос/ответ — "pong" от ::ffff:127.0.0.1 -> Unmap 127.0.0.1
```

Выводы, на которые опирается реализация:

1. **Unmapped v4 через dual-stack сокет доставляется** — отображение делает
   стандартная библиотека. Правило записи «отдавать как есть» подтверждено.
2. **Чтение обязано звать `Unmap()`.** v4-отправитель приходит как
   `::ffff:127.0.0.1` с `Is4() == false`. Без нормализации такой адрес **не
   сравнялся бы** с сохранённым чистым v4 — сломались бы `sameAddr`, дедуп
   `probes` и подтверждение endpoint'а. Это не гигиена, а условие работы.
3. Явная форма v4-in-v6 тоже доставляется — запасное правило записи существует,
   но не нужно.

`localEndpoints` (`internal/app/session.go:1358`) дополняется глобальными IPv6:
критерий `IsGlobalUnicast() && !IsPrivate()`, без link-local (`fe80::/10`) и ULA
(`fc00::/7`). STUN для них не запрашивается.

## Компоненты

### Фиксированный локальный порт

Сейчас порт случайный при каждом запуске (`internal/app/session.go:525`,
`Port: 0`). («Заморозка порта» из коммита `60be0cd` — про другое, там чинили
залипание STUN-рефлекса.) Это ломает две фичи: проброс делал бы новый маппинг при
каждом старте, засоряя таблицу роутера, а кэш endpoint'ов был бы наполовину
бесполезен — друзья помнят прежний `ip:port`, а узел уже на другом порту.

Порт хранится в `config.json`. Первый запуск: слушаем `Port: 0`, сохраняем
выданный. Дальше пробуем сохранённый.

Две неочевидные детали:

- **Порт выбирается из диапазона 20000–40000, а не из эфемерного.** Windows
  раздаёт динамические порты из 49152–65535; сохранённый оттуда порт после
  перезагрузки может оказаться занят посторонним приложением, обслуженным раньше.
- **Занятый порт → берём случайный, конфиг НЕ перезаписываем.** Иначе второй
  экземпляр (`run-node2.cmd`) при каждом старте угонял бы порт у первого.

CLI (`cmd/lanmesh`) получает флаг `-port` с дефолтом 0 — поведение консольного
клиента не меняется.

### `internal/portmap`

Каскад **PCP → NAT-PMP → UPnP-IGD**, первый ответивший выигрывает. Порядок
обоснован: PCP (RFC 6887) — единственный, который в принципе может работать
сквозь операторский CGN; NAT-PMP проще и быстрее; UPnP самый распространённый, но
самый тяжёлый (SSDP + SOAP + XML).

Реализация своя, без внешних зависимостей: тянуть `huin/goupnp` с деревом
UPnP-типов ради одной операции в проект, который поставляется одним
самодостаточным exe, несоразмерно.

Интерфейс не знает о lanmesh: «дай внешний `ip:port` для данного локального порта
и держи аренду, пока жив контекст».

```go
type Mapping struct {
    External netip.AddrPort
    Proto    string // "pcp" | "natpmp" | "upnp"
}

// Run держит маппинг живым до отмены ctx и снимает его при выходе.
// Обновления (в т.ч. смена внешнего адреса) приходят в канал.
func Run(ctx context.Context, localPort int) <-chan Mapping
```

Отбракованный маппинг (см. ниже) в канал не отдаётся вовсе — вызывающий не должен
повторять проверку у себя. Аренда обновляется на половине срока; при
выходе маппинг снимается, чтобы не засорять таблицу роутера. Единственное место с
syscall — определение шлюза через `iphlpapi.GetBestRoute` (нужно PCP и NAT-PMP;
UPnP обходится SSDP-мультикастом).

**Входящее правило брандмауэра — условие работоспособности, а не удобство.**
Сегодня правил нет вообще: `netsh` в проекте используется только для адаптера
(`internal/tun/tun_windows.go:180,191`), и этого хватает, потому что lanmesh всегда
пробивает **взаимно** — обе стороны шлют исходящие, и брандмауэр Windows
пропускает ответ как связанный.

Проброс порта создаёт случай, которого раньше не было. В паре cone↔CGNAT
cone-сторона слала на рефлексивный адрес `IP:портX`, а входящий приходит с
`IP:портY` — симметричный NAT выдал другой порт. Для брандмауэра это
**несвязанный входящий пакет**: роутер честно перешлёт, Windows выбросит. Без
правила проброс бесполезен ровно в том сценарии, ради которого затевался.

Поэтому при включённом пробросе заводится входящее правило на UDP-порт узла через
уже существующий помощник `runNetsh` (`internal/tun/tun_windows.go:163` — таймаут
и подавление окна консоли там уже сделаны):

```
netsh advfirewall firewall add rule name="lanmesh" dir=in action=allow \
      protocol=UDP localport=<порт> program=<путь к exe>
```

Правило привязано к `program=`, а не открывает порт всем желающим. Пересоздаётся
при смене порта и снимается вместе с маппингом при выключении проброса — иначе
после отказа от фичи в системе оставалось бы разрешающее правило, о котором
пользователь не знает.

**Обязательная отбраковка результата — без неё фича вредна.** Если полученный
внешний адрес приватный (RFC1918), из CGN-диапазона `100.64.0.0/10`, link-local,
**или** его IP расходится с тем, что видит STUN, — маппинг бесполезен (CGNAT или
двойной NAT) и **не анонсируется**. Иначе станет хуже, чем без проброса: пиры
будут долбиться в мусорный адрес вместо рабочих кандидатов. Наружу это выходит
честным статусом, а не молчанием.

### `internal/netcache`

Ключ — **пара (тег сети, PeerID)**: иначе адреса одной сети утекли бы в пробы
соседней. Хранятся **только подтверждённые прямые endpoint'ы** (те, с которых
пришёл расшифрованный кадр), не кандидаты — иначе кэш накопил бы мусор из DHT и
воспроизводил его при каждом старте.

TTL 30 дней, до 3 адресов на пира — **три последних по времени подтверждения**,
самый свежий вытесняет самый старый. Длинный TTL безопасен именно потому, что проба
стоит несколько UDP-пакетов с backoff, а мёртвый адрес отсеивается сам по
`probeTTL`.

Запись атомарная (temp + rename), по таймеру раз в минуту и при штатном выходе —
не на каждый пакет: файл не должен стать источником дисковой нагрузки под игровым
трафиком. Файл `endpoints.json` рядом с `config.json`.

**Не шифруется.** Обоснование: `config.json` уже хранит пароли сетей открытым
текстом (`cmd/lanmesh-gui/main.go:73`), поэтому шифрование соседнего файла с
адресами — театр: кто прочитал кэш, прочитает и пароли, и выведет из них всё
остальное. (Сам факт открытых паролей относится к подпроекту B, где переделывается
идентичность; здесь сознательно не трогается.)

Вливается через `AddProbes` при подключении сети — тот же путь, что у DHT.

### PEX — новый кадр `FramePeers byte = 8`

Кадр несёт **только список адресов, без PeerID.** Кто за ними окажется —
выяснится при расшифровке, ровно как с DHT. Это убирает и код сопоставления
идентификаторов, и саму возможность наврать про чужой ID.

- Отправка: подтверждённым пирам раз в 30 секунд, пиггибэком к `maintenance`.
- Содержимое: адреса **только подтверждённых прямых** пиров, не кандидатов —
  мусор не размножается.
- Потолок 16 записей на кадр, чтобы не раздувать пакет под MTU 1280.
- Приём: прямиком в `AddProbes` со всей его существующей защитой — дедуп,
  `maxProbes`, отдельный пул от `endpoints`.

Формат тела кадра — компактный, чтобы 16 записей с запасом влезали в MTU:

```
[count:1] затем count раз:
  [family:1] 4 = IPv4, 6 = IPv6
  [addr:4 или 16]  сетевой порядок байт
  [port:2]         big-endian
```

Максимум при 16 записях IPv6: 1 + 16×19 = 305 байт.

При неизвестном `family` разбор **останавливается**, но уже собранные адреса
сохраняются. Пропустить такую запись поштучно нельзя: её длина вычисляется из
семейства, и не зная семейства, непонятно, откуда читать следующую. Поля длины в
формате нет сознательно (YAGNI) — расширять протокол правильнее новым типом
кадра, который старые клиенты и так молча игнорируют.

Защита от вранья та же, что уже отработана на DHT: адрес становится пиром только
после расшифровки кадра ключом сети.

### Диагностика и UI

`cmd/natcheck` получает четвёртый шаг: попытка PCP/NAT-PMP/UPnP, вывод полученного
адреса, сравнение со STUN, наличие глобального IPv6. Это то, что можно разослать
друзьям и получить статистику вместо догадок.

В подробном режиме GUI — статус проброса («проброшен через PCP: 203.0.113.5:31337»
/ «роутер не поддерживает» / «за CGNAT, бесполезен») и наличие IPv6. Галка
«пробрасывать порт через роутер» в настройках: **включена по умолчанию**, потому
что фича бесполезна, если её никто не включит, а выигрывает от неё не тот, кто
включил, а его друзья за CGNAT. Состояние галки — поле в `config.json` рядом с
`port` (отсутствие поля читается как «включено», чтобы обновление у существующих
пользователей не выключало фичу молча).

## Поток данных

### Старт

Проброс **стартует в фоне и не ожидается**. Сейчас `bringUpNode` блокируется на
STUN (`internal/app/session.go:533`, до 3 с) и только потом создаёт адаптер;
добавлять каскад последовательно нельзя — время до появления сетевого адаптера
выросло бы вдвое. Результат проброса приходит поздно и вливается через
существующий механизм (`SetSelfCandidates` + `ReflexNotify` + `registerLoop`).

```
       ┌ STUN (3с, синхронно — как сейчас)      ─┐
свои ──┼ локальные интерфейсы v4+v6 (мгновенно) ─┼→ SetSelfCandidates → сигналка/DHT
       └ проброс (фоном, не ждём)               ─┘

       ┌ сигналка / DHT ─→ SyncPeers ─→ endpoints (потолок 12)
чужие ─┼ кэш прошлой сессии ─┐
       └ PEX от пиров        ─┴→ AddProbes ─→ probes (дедуп, потолок maxProbes)
                                                 ↓
                                   кадр расшифровался ключом сети
                                                 ↓
                                       learnPeerLocked → пир
```

Кэш вливается **при подключении сети, до ответа сигналки**: если адрес друга не
менялся, пробитие начинается на первой секунде, а не после первого раунда
регистрации.

### Выбор внешнего адреса при двух источниках

У узла появляется два внешних v4-адреса: рефлекс от STUN и проброшенный. В
signal-режиме проблемы нет — на сигналку уходит список кандидатов. Но в
**DHT-режиме анонсируется ровно один порт**, и `pickExternal`
(`internal/app/session.go:1156`) — код с историей: гистерезис там появился потому,
что при симметричном NAT relay-рефлекс и живой STUN видят разные порты и адрес
скакал каждый раунд.

Без правки `pickExternal` проброс в DHT-режиме не даст ничего. Правило:

| случай | решение |
|---|---|
| STUN промолчал | проброшенный адрес единственный — берём его |
| IP совпал со STUN, порт тот же | адреса равны, берём любой — поведение не меняется |
| IP совпал, порт другой | это симметричный NAT, где рефлекс врёт по определению → проброшенный выигрывает |
| IP не совпал | CGNAT/двойной NAT → не анонсируем вовсе (отбраковка в `portmap`) |

Побочный эффект первой строки: проброс становится вторым, независимым от STUN
источником внешнего адреса, и часть случаев жёлтого «ограничено» вылечится сама.

Гистерезис проброшенный адрес не ломает: он **стабилен по построению** —
endpoint-independent, один и тот же для всех адресатов. Он не участвует в скачках,
ради подавления которых гистерезис писался, а устраняет их.

## Обработка ошибок

Принцип: каждый источник отваливается **независимо и молча**, узел продолжает на
остальных. Ни один новый источник не может сделать хуже, чем сегодня.

| отказ | поведение | что видно |
|---|---|---|
| нет IPv6 у машины/провайдера | сокет фолбэком на `udp4` | «IPv6: нет» |
| правило брандмауэра не завелось (нет прав, политика домена) | проброс анонсируется, но несвязанные входящие дропаются — пара cone↔CGNAT остаётся на релее, как сегодня | «проброс: брандмауэр блокирует входящие» |
| роутер не отвечает на PCP/PMP/UPnP | каскад молча сдаётся | «проброс: роутер не поддерживает» |
| проброс вернул CGN/приватный адрес | не анонсируется | «проброс: за CGNAT, бесполезен» |
| аренда истекла, роутер забыл маппинг | обновление на половине срока; при провале адрес снимается из кандидатов | статус меняется |
| `endpoints.json` битый или чужой | читается как пустой, перезаписывается | ничего |
| адреса в кэше устарели | пробы дохнут по `probeTTL` | ничего |
| пир наврал в PEX | дедуп + `maxProbes`; кадр не расшифруется — пир не появится | ничего |
| роутер принял маппинг, но не пересылает | IP совпал со STUN → в signal-режиме лишний безвредный кандидат; в DHT-режиме выигрывает только там, где рефлекс и так не работал | — |

Последняя строка — единственный остаточный риск, и он ограничен конструкцией
правила выбора: проброшенный адрес вытесняет рабочий рефлекс только тогда, когда
рефлекс заведомо нерабочий.

## Тестирование

### Уровень 0 — доказательство отсутствия регрессии

Критерий «существующие тесты проходят без правок» **недостижим и не заявляется**:
тесты лежат в одном пакете с движком и обращаются к внутренностям напрямую
(`internal/peer/engine_test.go:300` присваивает `ps.active`;
`internal/peer/engine_test.go:169,191` и `relay_test.go:240` объявляют
`var active *net.UDPAddr`; `UseRelay` принимает `*net.UDPAddr` публично и так
вызывается из `discovery_test.go:198`, `relay_test.go:164`).

Гарантия переносится наружу:

1. **До единой правки в коде** пишется тест, гоняющий полный сценарий только через
   публичный API: два движка → пробитие → `FrameData` в обе стороны → ping/pong с
   RTT → фолбэк на релей при мёртвом direct → отбраковка повтора счётчиком →
   счётчики трафика. Внутренних полей не касается.
2. Фиксируется зелёным на текущем коде — это снимок сегодняшнего поведения.
3. После миграции обязан пройти **без единой правки**. Он и есть доказательство.

Для существующих тестов — правило, проверяемое на ревью: в
`git diff -- '*_test.go'` допустимы **только** смена типа переменной и способа
построения литерала адреса. Ни одного изменённого ожидания, порога, тайминга или
условия. Если под миграцию понадобилось поправить `if got != want` — миграция
изменила поведение; это стоп, а не повод подправить `want`.

То же с `pickExternal`: тесты в `internal/app/util_test.go:44,50,60,71` получают
шестой аргумент `""`, все существующие ожидания остаются дословно теми же.

### Уровень 1 — новые юнит-тесты

- **Нормализация адресов**: mapped на входе → unmapped внутри; v4, v6,
  link-local, некорректные.
- **`portmap`, отбраковка** — самый важный тест подпроекта: `100.64.1.1` →
  отбраковать, `192.168.1.1` → отбраковать, `169.254.x` → отбраковать,
  `203.0.113.5` → принять. Здесь ловится «стало хуже, чем без проброса».
- **`portmap`, разбор ответов**: бинарные PCP/NAT-PMP замокать; SOAP-ответ UPnP на
  записанном XML реального роутера.
- **`pickExternal`**: четыре новых случая правила выбора плюс все старые.
- **`netcache`**: TTL, потолок 3 адреса, битый файл → пустой, атомарность записи,
  изоляция по тегу (адрес из сети A не всплывает в пробах сети B).
- **PEX**: кодирование/декодирование кадра, потолок 16, в кадр попадают только
  подтверждённые прямые пиры.
- **Порт**: занят → берётся случайный и конфиг не перезаписан.

### Уровень 2 — интеграционные в Go

- **PEX**: три движка, A↔B и B↔C, A про C не знает. После раунда PEX у A
  появляется C — и появляется через расшифровку кадра, а не по слову B.
- **Кэш**: движок установил линк → адрес в кэше; новый движок из того же файла →
  адрес ушёл в `probes` до всякой сигналки.
- **IPv6**: два движка соединяются по `::1` — loopback v6 есть без сети, тест
  детерминированный.
- **v4-пир через dual-stack сокет** — обязателен, и вот почему. Характеризационный
  тест проверяет шаг 1 (миграция, сокет ещё `udp4`), тест выше проверяет чистый v6.
  А конфигурация, которая будет 90% продакшена — **v4-пир, разговаривающий через
  новый dual-stack сокет**, — не покрыта ни тем, ни другим и осталась бы только на
  ручной проверке. Тест: движок A на dual-stack сокете, движок B на `udp4`,
  `127.0.0.1`, полный цикл пробитие → данные → ping/pong.

### Уровень 3 — требует натурной проверки

Ни один пункт этого раздела не может быть проверен в сессии разработки и не должен
считаться выполненным без вывода с живой машины.

| проверка | как | что доказывает |
|---|---|---|
| **cone ↔ CGNAT** | ПК дома + вторая машина через телефон-точку; прогнать дважды — с выключенным пробросом и с включённым | главный тест подпроекта: 🔵 релей → 🟢 прямой. Если этого не произойдёт, фича 1 не окупилась |
| проброс на живом роутере | `natcheck` с новыми пробами | какой из трёх протоколов сработал, совпал ли со STUN |
| нативный IPv6 | `natcheck` дома и на телефоне | есть ли смысл в IPv6 у этой группы |
| время до адаптера | замер до и после изменений | «функционал не ухудшился» буквально |
| v4-пробитие цело | обычная сеть с другом | регрессия от dual-stack |
| PEX по-настоящему | собраться втроём, затем выключить сигналки в настройках — сеть обязана жить | фича 3 |
| кэш | собраться, закрыть, поднять снова, засечь время до линка | фича 4 |
| статистика | разослать `natcheck` друзьям, собрать вывод | скольким реально помогает |
