# lanmesh GUI — редизайн «Command Glass»

**Дата:** 2026-07-18
**Ветка:** `gui-redesign`
**Статус:** дизайн согласован, готов к разбивке на план имплементации (Phase 1)

---

## 1. Контекст и проблема

`cmd/lanmesh-gui` — это Go-приложение, которое встраивает один файл `index.html`
(`go:embed`), поднимает локальный HTTP-API на `127.0.0.1:8737` и открывает его в
окне **WebView2** (сейчас 430×640, min 360×480). «GUI» = веб-фронтенд внутри
нативного окна.

Проблема: интерфейс «совсем никуда» — плоский, служебный, использует лишь малую
часть уже доступных данных. Бэкенд отдаёт по каждому пиру статус
(`direct`/`relay`/`connecting`), RTT, «когда последний раз видели», видимость на
сигналках; по узлу — внешний адрес, тип NAT (`/api/diagnose`), аптайм. Почти
ничего из этого не показано выразительно.

Хорошая новость: технологию менять **не нужно** — весь визуал живёт во фронтенде.

## 2. Цели и не-цели

**Цели:**
- Новый визуальный язык — тёмное «стекло» с неоновым акцентом (см. §4).
- Два режима в одном приложении: **компактный** (быстрый взгляд, узкое окно) и
  **подробный** (Sidebar Dashboard с картой, графиками, деталями).
- Живые фичи: график пинга (sparkline) и индикатор качества по каждому участнику;
  далее — трафик RX/TX, карта сети, онбординг, in-app уведомления.
- Сохранить **всю** текущую функциональность и защиту (санитизация имён, CSRF-guard).

**Не-цели:**
- Не менять сетевое ядро/крипто/сигналинг.
- Не вводить фронтенд-фреймворк и сборочный тулчейн (см. решение §5.2).
- Не переезжать с WebView2 на нативный тулкит.

## 3. Итоги брейншторма (согласовано с пользователем)

- Визуальное направление: **Command Glass** — тёмная база + эффекты стекла
  (насыщенный вариант: аврора-свечение, морозный blur, графики пинга).
- Раскладка: **оба режима** — компактный и максимально подробный.
- Компактный режим: **без аватарок**, узкие вертикальные карточки-строки.
- Подробный режим: аватарки и графики уместны, оставляем богаче.
- Приоритет живых фич: пинг+качество, трафик RX/TX, карта сети, онбординг+трей.

## 4. Визуальный язык — токены «Command Glass»

```css
:root {
  /* поверхности */
  --bg-page:   #08090c;
  --bg-panel:  #0a0d12;
  --glass:        rgba(255,255,255,.06);
  --glass-border: rgba(255,255,255,.11);
  --glass-hi:     rgba(255,255,255,.09);   /* inset top highlight */
  --blur: 16px;                            /* backdrop-filter: blur() saturate(1.2) */
  /* аврора-свечение (низкая непрозрачность, база остаётся тёмной) */
  --glow-mint:   rgba(52,229,176,.18);     /* верх-лево  */
  --glow-violet: rgba(139,92,246,.20);     /* низ-право  */
  /* акценты */
  --accent:      #37f5c5;  --accent-2: #46e6c0;  /* direct / бренд (неон-мята) */
  --relay:       #8fbaff;                        /* relay */
  --ping-ok:     #ffcf5c;                        /* средний пинг / connecting */
  --ping-bad:    #ff8080;                        /* высокий пинг */
  /* текст */
  --fg:      #e2e7ef;  --fg-mut: #6f7a8a;  --fg-dim: #57616f;
  /* форма */
  --r-card: 14px;  --r-row: 12px;  --r-pill: 999px;
  --mono: ui-monospace, "Cascadia Code", Consolas, monospace;
}
```

- **Пороги пинга** (сохраняем текущие из `index.html`): good `<60`, ok `<150`,
  bad `>=150` мс. Цвет пинга и цвет точки-статуса — по этим порогам/статусу.
- **Статус-точка:** `direct` → мята со свечением, `relay` → синяя, `connecting`
  → жёлтая (пульс).
- Числа (IP, пинг, трафик) — моноширинные, `font-variant-numeric: tabular-nums`.
- Имена пиров/сетей — только через `dispName()` (см. §7): экранирование +
  вычистка bidi/zero-width/управляющих символов. **Регресс тут недопустим.**

Референс-макеты (согласованы): `visual-direction`, `dark-glass`, `layout`,
`two-modes`, `compact-v2` (собраны в брейншторме; воспроизводятся кодом Phase 1).

## 5. Архитектурные решения

### 5.1 Транспорт UI — оставляем WebView2 + локальный HTTP-API
Альтернатива — нативный тулкит (Wails/Fyne) или замена рендерера. Отклонено:
весь дефицит качества — во фронтенде; смена стека = максимум работы и рисков без
выигрыша в визуале. Один Go-бинарь и текущая модель «сервер+webview» сохраняются.

### 5.2 Фронтенд — организованные статические ассеты, ванильный JS, БЕЗ сборки
Три варианта:
- (A) один большой `index.html` — не масштабируется на несколько видов;
- (B) **набор статических ассетов** (`index.html` + `app.css` + ES-модули JS),
  раздаются через `go:embed` каталога; **без npm/сборки** — выбрано;
- (C) SPA-фреймворк + сборка (Vite/Svelte) — отклонено: добавляет Node-тулчейн в
  Go-проект, усложняет сборку exe и порог входа контрибьютора. Проект ценит
  «один бинарь без тулчейна» — сохраняем это.

Карта и графики рисуются **инлайн-SVG/Canvas без библиотек**. WebView2 (Edge
Chromium) поддерживает ES-модули по `http://` — раздаём модули из встроенного FS.

### 5.3 Два режима — CSS-отзывчивость + ручной тумблер
- Phase 1: режим определяется **шириной контейнера** (ResizeObserver/container
  query) и ручной кнопкой ⤢, которая ставит класс `compact`/`detailed` и
  запоминает выбор (localStorage). Нативный вызов не нужен.
- Phase 2: кнопка ⤢ дополнительно меняет **размер нативного окна** через
  `webview2.SetSize` (есть в библиотеке). Мост JS→native (`webview2.Bind`) на
  этой машине не проверен (Go не установлен) — **проверить при имплементации
  Phase 2**; если `Bind` недоступен, окно остаётся ресайзабельным вручную, а
  тумблер меняет только раскладку.

## 6. Информационная архитектура

**Компактный режим** (окно ~300–360 px):
`header (бренд · статус-pill · ⤢)` → `self-строка (IP · внешний · аптайм)` →
`карточки сетей` (каждая: заголовок сети + счётчик + сигнал-точки + Пригласить/
Выйти) → `узкие строки участников` (точка-статус · имя · IP · пинг) →
`форма «＋ Добавить сеть»` → `сервера/диагностика (сворачиваемо)`.

**Подробный режим** (окно ~960×620, resizable):
- **Рейл слева:** бренд; список сетей (имя + «онлайн/всего», активная подсвечена);
  навигация **Список / Карта / Трафик / Настройки**; снизу мини-статус узла.
- **Основная область:** заголовок текущей сети (+ ⤡ Компактно, Пригласить,
  Выйти) → плитки-статы (Качество; Трафик — с Phase 3) → богатые строки
  участников (аватар · имя · IP · сигнал-бары · sparkline пинга · пинг · статус).

Виды **Карта** и **Трафик** в Phase 1 присутствуют в навигации как заглушки
(«скоро»); наполняются в Phase 3/4.

## 7. Функциональность к сохранению (без регресса)

Все текущие эндпоинты и поведение переносятся 1:1:
`GET /api/state`, `POST /api/addnetwork` (network, password, signals?, relay?),
`POST /api/leavenetwork` (tag), `POST /api/disconnect`, `POST /api/sendlogs`,
`POST /api/senddiag`, `GET /api/diagnose`, `GET|POST /api/settings`,
`GET /api/invite?tag=`.

Обязательно сохранить:
- Разбор приглашения `lanmesh://join?net=&pass=&sig=&relay=` (текстом, без
  регистрации URL-протокола) и автозаполнение полей.
- `dispName()` -- экранирование HTML + вычистка управляющих, bidi-override и zero-width символов из имён (точный набор диапазонов -- как в текущем `index.html`; иначе именем можно визуально подделать другой узел).
- Логику мультисети, «сервера правим только когда узел снят», скрытие адресов
  серверов из UI, заметки о приёме серверов из приглашения.
- Поведение «форма добавления раскрыта, пока сетей нет».

## 8. Живые фичи и источник данных

| Фича | Источник | Фаза |
|---|---|---|
| Sparkline пинга по пиру | фронт копит историю RTT из `/api/state` (ключ — vip) | P1 |
| Индикатор качества | композит: статус (direct/relay) + текущий RTT + стабильность (дисперсия RTT) | P1 |
| Сигнал-бары | `peer.signals[]` (видимость на сигналках) | P1 |
| NAT/диагностика в UI | `GET /api/diagnose` (уже есть, сейчас скрыт) | P2 |
| In-app тосты «друг зашёл/вышел» | дифф состава пиров между опросами (фронт) | P2 |
| Трафик RX/TX | **новые** счётчики в движке (`tunToNet`/`netToTun`, поля `peerState`) → `PeerView`/`StateView` | P3 |
| Карта сети (SVG) | `state.networks[].peers[]` | P4 |
| OS-трей + нативные уведомления | новый код (systray + Windows toast) | P5 |

## 9. Фазовый план и обоснование декомпозиции

Работа большая и разнородная; риск и зависимости изолируем по фазам
(осознанное решение, не «чтобы быстрее»): фронтенд не зависит от Go, а самый
рискованный кусок (трей vs. main-thread WebView2) не должен держать редизайн в
заложниках.

- **P1 — Визуальный каркас** *(без Go)*: новый фронт + mock-харнесс; тёмное
  стекло, оба режима, список участников, sparkline пинга, индикатор качества,
  вся текущая функциональность. **Здесь основная ценность.** ← первый план.
- **P2 — Интеграция в Go**: `go:embed` новых ассетов, размер/ресайз окна,
  показ диагностики, in-app тосты.
- **P3 — Трафик**: счётчики RX/TX в движке → вид «Трафик».
- **P4 — Карта сети**: SVG-топология.
- **P5 — OS-трей + нативные уведомления**: отдельная спека (риск main-thread).

Каждая фаза = свой цикл план → имплементация → проверка.

## 10. Phase 1 — детальная спецификация (готово к плану)

**Объём:** только фронтенд-ассеты + dev-харнесс. Правок Go в P1 нет.

**Компоненты и файлы (раскладка ассетов, встраиваются в P2):**
- `index.html` — разметка-оболочка (шапка, контейнеры режимов, слоты видов).
- `app.css` — токены §4 + компоненты (pill, glass-row, badge, bars, sparkline,
  rail, tiles, формы, состояния ошибок/пусто).
- `app.js` — опрос `/api/state` (интервал ~1.2–1.5 с), оркестрация рендера,
  переключение режимов, история RTT, дифф пиров.
- `views/list.js` — рендер списка (компакт + подробный).
- Заглушки `views/map.js`, `views/traffic.js` («скоро»).
- Хелперы: `esc`/`dispName`, `fmtRtt`/`fmtUptime`/`fmtSeen`, `plural`, `parseInvite`
  (переносятся из текущего `index.html`, покрываются переносом поведения).

**Состояния, которые обязаны рендериться корректно (проверяем все):**
не подключено; узел без внешнего адреса (warn); ошибка сигналки (degraded);
0 участников (пустая сеть с подсказкой-приглашением); 1 сеть; несколько сетей;
пиры `direct`/`relay`/`connecting` вперемешку; высокий пинг; переключение
компакт↔подробный; **враждебное имя** (bidi/zero-width — проверка санитизации).

**Живые фичи P1:**
- История RTT: кольцевой буфер (~30–60 точек) на пира (ключ vip), sparkline
  инлайн-SVG; сброс при исчезновении пира.
- Качество: `direct` + низкий стабильный RTT → «хорошее»; `relay` или высокий/
  нестабильный RTT → «среднее»/«плохое»; отражается цветом и подписью.

**Явно вне P1:** любые правки Go; реальные счётчики трафика; отрисовка карты;
OS-трей. Плитка «Трафик» и вид «Карта» — заглушки.

## 11. Стратегия проверки (evidence, не на словах)

Go на машине **не установлен** → основной инструмент проверки Phase 1 —
**mock-харнесс** (Node v24 доступен): маленький статический сервер, который
раздаёт ассеты фронтенда и отвечает на `/api/state`, `/api/diagnose`,
`/api/settings`, `/api/invite` фейковыми данными, **перебирая сценарии** из §10.
Проверка = открыть в браузере (тот же движок Chromium, что и WebView2), пройти
все состояния и оба режима, глазами подтвердить рендер; отдельно — тест
санитизации имени.

Для последующих фаз (интеграция, счётчики) понадобится Go — установим, когда
дойдём до P2/P3, либо сборку/прогон делает пользователь. Юнит-тесты движка
(`internal/peer`) покрывают счётчики трафика в P3.

## 12. Риски и открытые вопросы

- **P5 (трей):** OS-трей-иконка хочет тот же message-loop, что и `w.Run()`
  WebView2 на Windows — нетривиально. Ценность уведомлений частично закрываем
  раньше in-app тостами (P2). Трей — отдельная спека со своим спайком.
- **`webview2.Bind`** для ресайза из UI не проверен (нет Go) — решается в P2;
  фолбэк — ручной ресайз (§5.3).
- **Go не установлен** — не блокирует P1; блокирует прогон Go-частей до установки.
