# lanmesh

Свой аналог Radmin VPN: виртуальная локальная сеть для игр с друзьями поверх
прямых P2P-туннелей. Модель доступа как у Radmin — **имя сети + пароль**, никаких
аккаунтов. Трафик шифруется end-to-end (ChaCha20-Poly1305), сервер-сигналка видит
только сведение пиров, но не сам трафик.

## Как это устроено

```
   [клиент A] ── UDP hole punch ──> [клиент B]      (прямой P2P, минуя сервер)
        │                                │
        └────────> [Cloudflare Worker] <─┘          (только сведение: кто в сети + endpoint'ы)
                       сигналка
```

- **Виртуальный адаптер** (Wintun) с адресом `25.x.y.z/8`, выведенным из ID узла.
  Для игр это выглядит как обычная локалка.
- **Шифрование**: ключ сети = Argon2id(пароль, соль=имя сети). Все, кто знает
  имя+пароль, получают один ключ; сервер его не видит.
- **Пробитие NAT**: клиент узнаёт свой внешний адрес через STUN и обменивается
  им через сигналку, затем обе стороны шлют друг другу UDP — открывается дырка.
- **Broadcast для LAN-игр**: широковещательные пакеты (поиск серверов в локалке)
  дублируются юникастом всем пирам, поэтому старые игры видят друг друга.

## Компоненты

| Путь | Что |
|------|-----|
| `cmd/lanmesh` | клиент (CLI) |
| `internal/crypto` | вывод ключа из имени+пароля, шифрование пакетов |
| `internal/proto` | идентичность узла, виртуальный IP, типы сообщений |
| `internal/signal` | клиент сигналки + минимальный STUN |
| `internal/peer` | движок mesh: маршрутизация, пробитие NAT, broadcast |
| `internal/tun` | обёртка Wintun (Windows) |
| `internal/app` | переиспользуемый сеанс сети (Start/Stop/State) |
| `cmd/lanmesh-gui` | графическая оболочка: веб-панель + системный трей |
| `internal/logbuf` | кольцевой буфер лога для удалённой диагностики |
| `worker/signal-worker.js` | сигнальный сервер на Cloudflare Worker |
| `cmd/natcheck` | диагностика: какие STUN доступны и какой тип NAT |

## Сборка

```
go build -o lanmesh.exe ./cmd/lanmesh
```

Рядом с `lanmesh.exe` нужен `wintun.dll` (уже лежит; оригинал — https://www.wintun.net).

## Сервер-сигналка

Сигналок **две, и клиент ходит в обе сразу**, сливая списки участников по
peerID. Это не «основная и запасная»: при переключении сеть раскололась бы —
одни на одной сигналке, другие на другой, друг друга не видят, и обе стороны
считают, что всё в порядке. Достаточно, чтобы у двух узлов нашлась хоть одна
общая живая.

| Адрес | Где | Зачем |
|---|---|---|
| `https://your-worker.example.workers.dev` | Cloudflare Worker | не зависит от домашнего интернета |
| `https://your-server.example.com:25557` | Orange Pi, `cmd/lanmesh-signal` | у части провайдеров DPI режет `*.workers.dev` |
| `http://your-server.example.com:25556` | она же, плайнтекст | только ради клиентов старых сборок |

Всё это дефолты клиента — флаг `-signal` можно не указывать. Секретов не нужно:
сервер знает лишь несекретный тег сети и endpoint'ы, ключа сети у него нет и
трафик он расшифровать не может.

Плайнтекстовый 25556 подлежит отключению, когда все обновятся: по HTTP тег сети
виден по дороге, а тег — это доступ к чужой диагностике через `/logs`.

### TLS на своей сигналке

Сертификат — Let's Encrypt на `your-server.example.com`, обновляет `certbot`
(webroot `/var/www/html`, отдаётся на 80-м порту). Сервису он приходит через
`LoadCredential=` в юните: certbot кладёт ключ под root, а сервис работает с
`DynamicUser=yes` (UID каждый раз новый, `chown` невозможен), поэтому systemd
читает файлы рутом до сброса привилегий и кладёт копии в приватный каталог.

⚠️ Это **снимок на момент старта** — сам он не перечитается. После обновления
сертификата сервис перезапускает хук
`/etc/letsencrypt/renewal-hooks/deploy/restart-lanmesh-signal.sh`. Перезапуск
дёшев: состояние в памяти, пиры перерегистрируются за 20с, а клиент терпит
осечку сигналки до 90с (`peerForget`).

⚠️ Обновление сертификата завязано на **Pi-hole**: webroot `/var/www/html`
отдаёт наружу именно `pihole-FTL` на 80-м порту. Если Pi-hole убрать, обновление
молча сломается, и TLS протухнет в течение месяца — тогда certbot надо
переключить на `--standalone`.

### Передеплой воркера

После правок `signal-worker.js` (нужен API-токен Cloudflare в `worker/.cf_token`):

```
cd worker
$env:CLOUDFLARE_API_TOKEN = (Get-Content .cf_token -Raw).Trim()
npx --yes wrangler@latest deploy
```

Состояние воркера живёт в Durable Object (`MeshRegistry`, по объекту на сеть) —
KV не подошёл: на бесплатном тарифе 1000 записей в сутки, а регистрация раз в
20с даёт ~4300 на КАЖДЫЙ узел. Токен деплоя лежит в `worker/.cf_token` (в
`.gitignore`, шаблон «Edit Cloudflare Workers»). Деплой ходит через
корпоративный прокси штатно; интерактивный `wrangler login` на этой машине НЕ
работает (браузер не достаёт callback на localhost через прокси) — только токен.

## Запуск (у каждого друга)

**Графический вариант (рекомендуется):** двойной клик по `lanmesh-gui.exe`. Windows
попросит права администратора (нужны для сетевого адаптера), после чего в браузере
откроется панель управления `http://127.0.0.1:8737` и появится иконка в трее.

> `lanmesh-gui.exe` самодостаточен: `wintun.dll` вшит внутрь и распаковывается
> рядом с exe при первом запуске. Другу достаточно отправить **один файл** —
> положить в папку с правом записи (Загрузки/Рабочий стол) и запустить. Т.к. exe
> без цифровой подписи, SmartScreen покажет «Windows защитила ваш компьютер» →
> «Подробнее» → «Выполнить в любом случае»; антивирус тоже может переспросить.
В панели ввести имя сети и пароль → «Подключиться». Галочка «запомнить» включает
автоподключение при следующем запуске. Панель показывает твой виртуальный IP и всех
участников сети с их статусом (🟢 прямое соединение / 🟡 идёт пробитие NAT).

**Консольный вариант** (без интерфейса), из-под администратора:

```
lanmesh.exe -network myteam -password hunter2
```

Сигналка подставляется по умолчанию; свою можно задать через `-signal`.

### Сборка GUI

```
go build -ldflags "-H=windowsgui" -o lanmesh-gui.exe ./cmd/lanmesh-gui
```

Флаг `-H=windowsgui` убирает консольное окно. Логи пишутся в
`%APPDATA%\lanmesh\gui.log`, настройки — в `%APPDATA%\lanmesh\config.json`.

Всем участникам — **одинаковые** `-network` и `-password`. Через несколько секунд
появляются виртуальные адреса `25.x.y.z`; проверить связь — `ping 25.x.y.z` до
друга, дальше запускать игру и искать сервер по LAN или по этому адресу.

### Подключение по IP:порт (Minecraft и т.п.)

Туннель работает на уровне IP-пакетов и прозрачен для портов и протоколов
(TCP/UDP) — как обычная локальная сеть. Пример с Minecraft: хост поднимает сервер
(слушает `0.0.0.0:25565` по умолчанию), остальные в «Direct Connect» вводят
`<виртуальный-IP-хоста>:25565`, например `25.31.208.7:25565`. Так же работает любое
приложение на любом порту. Автопоиск «Open to LAN» (мультикаст-анонс) тоже
проходит через broadcast-эмуляцию, но надёжнее подключаться напрямую по адресу.

MTU виртуального адаптера занижен до 1280 под накладные расходы туннеля (в худшем
случае — путь через ретранслятор, +49 байт relay-заголовка) — без этого крупные
пакеты (чанки, файлы) фрагментировались бы и терялись, особенно через relay и на
мобильном LTE/CGNAT, а соединение вроде «в сети», но по факту виснет.

## Ретранслятор (relay)

Прямое соединение открывается не всегда. За **симметричным NAT** — а это, в
частности, любой мобильный интернет с CGNAT — внешний порт меняется для каждого
адресата, и пробитие невозможно в принципе. Для таких пиров есть запасной путь:
трафик идёт через сервер на белом адресе.

```
cmd/lanmesh-relay        сервер (Linux/arm64, systemd на orangepi)
your-server.example.com:25555   боевой адрес (дефолт клиента, флаг -relay)
```

Протокол (UDP):

```
клиент -> relay   [0x01][тег(32)][peerID(16)]              bind/keepalive
клиент -> relay   [0x02][тег(32)][dstPeerID(16)][кадр...]  переслать узлу dst
relay  -> клиент  [0x03][кадр...]                          входящий кадр
relay  -> клиент  [0x04][тег(32)][peerID(16)]              подтверждение bind
```

Ретранслятор **ничего не расшифровывает**: внутри лежит обычный запечатанный
кадр, ключа сети у сервера нет. Он видит только несекретный тег и адреса — как и
сигналка. Таблица ключуется парой (тег, peerID), так что посторонний не подменит
запись в чужой сети, не зная пароля.

Логика клиента: сначала `relayGrace` (6с) честно пробуем пробиться, и только
потом идём в обход. Кандидаты долбим **и дальше** — если дырка откроется, движок
сам переключится на прямой путь: он быстрее и не грузит чужой сервер. Пакет,
пришедший через relay, НЕ подтверждает прямой endpoint (иначе движок решил бы,
что пробился, и слал бы данные на адрес ретранслятора как на прямой).

В панели такой пир помечен «🔵 ретранслятор».

### Развернуть свой

```
GOOS=linux GOARCH=arm64 CGO_ENABLED=0 go build -o lanmesh-relay-arm64 ./cmd/lanmesh-relay
```

Закинуть в `/usr/local/bin/lanmesh-relay`, положить unit в
`/etc/systemd/system/lanmesh-relay.service`, `systemctl enable --now`. На роутере
нужен проброс **UDP** (не TCP!) порта 25555 на этот хост — и лучше закрепить за
ним статический адрес, иначе однажды DHCP выдаст другой и правило будет вести в
пустоту.

## Пинг между участниками

Панель показывает задержку до каждого пробитого пира (зелёный <60 мс, жёлтый
<150 мс, дальше красный). Работает так: подтверждённому пиру раз в 5 секунд
уходит `FramePing` с 8-байтным номером, тот возвращает его эхом в `FramePong`,
и отправитель считает RTT **по своим часам** — синхронизация времени между
машинами не нужна и не предполагается. Ответ с чужим номером игнорируется,
иначе потерянный ping завышал бы задержку.

Ping заодно работает keepalive'ом для подтверждённых пиров (отдельного больше
нет): это такой же UDP-пакет в тот же endpoint, дырку в NAT он держит не хуже.
Непробитые пиры по-прежнему долбятся `FramePunch` раз в 2 секунды.

> Замер в локалке честно бывает **нулевым**: задержка там меньше зернистости
> монотонных часов Windows (~1 мс). Поэтому «измерено» определяется по времени
> замера, а не по `rtt > 0` — иначе самые быстрые соединения показывали бы «—».

## Диагностика чужой машины

Чтобы разбирать «у меня не подключается» без пересылки файлов, клиент умеет
складывать свой лог на сигналку, а забрать его можно одной командой.

В панели есть галка **«отправлять диагностику на сигналку»** (включена по
умолчанию, состояние хранится в `config.json`). Раз в 30 секунд накопленные
строки уходят в `POST /log` — но **только если они есть**: запись в KV не
бесплатна, а лог почти всегда молчит.

Забрать логи всей сети. Нужен тег сети — несекретный идентификатор
`sha256("lanmesh-tag|"+ключ)`; считается локально из имени+пароля:

```
lanmesh.exe -network myteam -password hunter2 -tag
curl "https://your-worker.example.workers.dev/logs?net=<тег>"
```

Отдаётся простым текстом, по строке на запись, с именем узла в начале. Логи
живут **час** (`LOG_TTL`), потом исчезают сами.

Кто может прочитать: тот, кто знает тег, то есть имя+пароль сети — ровно те же
люди, что и так видят через `/register` хостнеймы и endpoint'ы друг друга.
Паролей в логах нет, но есть хостнейм, локальные адреса и внешний IP.

## Определение внешнего адреса (STUN)

Клиент опрашивает **сразу несколько** STUN-серверов разных операторов
(`signal.DefaultSTUNServers`) с того же сокета, что слушает туннель, и берёт
первый ответивший. Одним сервером обходиться нельзя: если провайдер режет его
(например, Google), узел остаётся без внешнего адреса, и пробить к нему дырку
невозможно в принципе — он просто невидим для остальных. Опрос идёт разом, а не
по очереди, поэтому полное молчание стоит один таймаут (3с), а не N×3с.

Если внешний адрес получить не удалось, панель показывает жёлтое
**«ограничено»** и объясняет причину — раньше в этом случае она врала зелёным
«подключено», хотя участник был недостижим.

Проверить, что доступно с конкретной машины, и заодно тип NAT:

```
go run ./cmd/natcheck
```

## Статус

Работает end-to-end: идентичность, STUN, Wintun-адаптер, шифрованный транспорт,
пробитие NAT, broadcast-эмуляция, сигналка. Проверено: адаптер поднимается, STUN
возвращает внешний endpoint, крипто-раунд-трип покрыт тестами. **Сигналка
задеплоена и протестирована** (регистрация + сведение двух пиров работают).
Осталось: реальный тест туннеля между двумя машинами.

### Ограничения / что дальше

- **Только Windows** (Wintun). Linux/macOS — добавить `internal/tun` под их TUN.
- **Нет relay-фолбэка**: если оба за симметричным NAT/CGNAT, дырка не пробьётся и
  пир останется недоступным. План — ретранслятор на orangepi (у него белый вход).
- **Full mesh**: каждый шлёт каждому. Для десятка друзей ок; на сотни не рассчитано.
- Виртуальный IP выводится из ID узла — теоретически возможна коллизия адресов
  (крайне маловероятна для малых групп); при желании разруливается на сигналке.
